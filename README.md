# AI OX-Plow ( AI - Engineering )

AI의 실수와 미스 캐치 등을 줄이기 위한 여러 가지 방법을(프롬프트, 컨텍스트, 하네스 엔지니어링) 적용해 봤지만

결국 수많은 제재와 동일한 명령 반복 추가 기입 그리고 프롬프트 한 줄 수정할 때마다 다시 많은 테스트 필요 등

실무에 사용하기까지 꽤 공수가 필요하고 사용하더라도 자율성으로 인한 결과물이 매번 틀려져 신뢰도가 떨어짐

## OX-Plow
**AI의 자율성을 유지한 채 오류를 허용하고, 상태 기반 반복과 누적 개선을 통해 결과를 지속적으로 정제·검증( 묵묵히 느리지만 단단하게 )**

### 1. AI가 비슷한 업무를 주기적으로 반복하도록 구성.
  - 프롬프트 + 컨텍스트 + 하네스 등 적당히 적용

### 2. AI의 실수나 오류에도 반복 지속으로 전체 흐름을 유지하도록 구성.
  - 주기적으로 정해진 범위를 조금씩 앞으로 롤링 등( 포인트 스케일링 또는 페이퍼 롤링 등 )

### 3. AI와 주기별 상태를 관리하고 결과물을 누적 개선해, 최종 결과만 확인하도록 구성.
  - 해결/미결 등의 상태 관리나 중간 제외, 체크 추가 등 AI와 함께 만든 최종 결과를 휴먼 확인

예로 지난 패치 또는 오늘부터 5일 전 이후 등 매일 코드 리뷰(특정 이슈 제외 또는 체크 등 유동적으로 추가)를 시켜 이슈 누적 리스트업

## 적용 사항
### **선악과의 궁금증은 참을 수 없다 -> 물리적 스탭이나 파일 분할 등 고려**

#### 1. 흐름 저장 
  - 결과물을 한 번에 만들지 않고, 파일이나 메모리 등으로 관리하여 어떤 상황에서도 흐름을 이어나갈 수 있게 구성

#### 2. 스탭 분할 
  - 컨텍스트 오염과 불필요 참조, 한 작업 장시간 실행 방지 그리고 단위 디버깅 등를 위해 모듈 단위로 외부에서 여러 번의 스탭으로 실행할 수 있게 구성

#### 3. 참조 제어
  - 과거 파일 참조 컨트롤( 예로 너무 오래되거나 진행에 불필요한 리포트는 삭제나 분리 보관하여 참조를 원천 차단하여 컨텍스트 오염 방지 )

### **한번에 모든걸 할 수 없다 -> 다수의 반복과 검증 고려**

#### 4. 반복 누적 
  - 매번 반복되는 동안 결과물들이 쌓이고 폴리싱되어 퀄리티를 증가하게 구성( 예로 중요도나 상태 등으로 이슈 관리 )

#### 5. 결과 갱신 
  - 잘못 된 결과물을 제외 또는 해결되거나 오래되거나 신뢰가 낮은 항목들은 다음 주기에서 배제하여 결과물 양을 관리할 수 있도록 구성

#### 6. 품질 검증 
  - 최종 품질 검증 및 폴리싱 및 테스트 단계 추가

#### 7. 휴먼 관리 
  - 반복 주기에 휴먼이 수동 관리할 수 있도록 구성해서 AI의 실수나 미스 매치 등 제어( 예. 장기 추적을 위한 결과물 포함과 결과물에서 제외 항목 등 관리 )

## 기타 사항

### 다양한 모델 적용  
  - 주기 별 또는 모듈 별 AI 모델 변화로 교차 검증으로 여러 의견 수렴( 예로 일간 리포트의 경우 평일 낮은 AI 모델 사용, 주말엔 비싼 AI 모델 투입 )

사용 예) 자동 코드 리뷰, CS/QA 자동 댓글, 월 간 데이터 통계 등등
